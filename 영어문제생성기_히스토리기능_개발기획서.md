> **작성일**: 2026-03-30
> **범위**: 세션 저장/복원(히스토리) 기능 추가
> **원칙**: 브라우저 저장 우선 구현, 추후 백엔드 동기화 가능하도록 설계

## 1. 프로젝트 개요

### 1-1. 프로젝트 소개

한국 중·고등학교 영어 교사를 위한 **시험 문제 자동 생성 웹 애플리케이션**이다. 5가지 유형의 영어 문제를 AI(OpenAI API)와 로컬 로직을 활용하여 자동 생성하고, Word(.docx) 파일로 내보낼 수 있다.

### 1-2. 기술 스택

| 구분 | 기술 |
|------|------|
| 프레임워크 | React 19 + Vite 8 |
| 스타일 | Tailwind CSS 4 |
| 라우팅 | React Router DOM 7 (HashRouter) |
| 상태 관리 | React Context API (`AppContext`, `SettingsContext`) |
| AI API | OpenAI — `gpt-5.4-mini` (문제 생성), `gpt-5.2` (Vision/OCR), `gpt-5.4` (지문 변형 2-Pass) |
| 문서 생성 | `docx`, `docxtemplater`, `mammoth`, `file-saver` |
| 배포 | GitHub Pages (100% 클라이언트 사이드 SPA) |

## 2. 현재 시스템 분석

### 2-1. 5가지 문제 유형별 데이터 흐름

#### (1) 누적 시험지 (`/cumulative` — `CumulativeTest.jsx`)

```
Word 파일 업로드 → mammoth 파싱 → 과별 단어 추출
→ 50단어 테스트 조합 (부족분은 이전 과 보충)
→ docxtemplater로 template.docx 기반 Word 내보내기
```

- **AI 사용**: 없음
- **저장 대상 데이터**: 업로드된 파일 정보, 파싱된 과별 단어, 생성된 테스트 구성

#### (2) 어휘 기반 문제 (`/` — `TypeB_Blank.jsx`)

```
단어 입력(텍스트/이미지) → AI 파싱(OCR) → parsedWords 배열 생성
→ 유형별 문제 생성 (Type1: 로컬 / Type2,3: AI API)
→ questions 배열 → docx 라이브러리로 Word 내보내기
```

- **AI 사용**: Type2(빈칸매칭), Type3(단일빈칸) — 예문 생성
- **저장 대상 데이터**: `rawText`, `parsedWords`, `difficulty`, `typeCounts`, `questions`

#### (3) 지문 기반 문제 (`/passage` — `TypeA_Passage.jsx`)

```
단어 입력 + 지문 입력 → AI 파싱 → parsedWords + passages
→ AI가 지문 내 단어 매칭 문제 생성
→ questions 배열 → Word 내보내기
```

- **AI 사용**: 문제 생성
- **저장 대상 데이터**: `rawText`, `parsedWords`, `passagesText`, `passageImage`, `difficulty`, `typeCounts`, `questions`

#### (4) 독해 O/X (`/reading-ox` — `ReadingOX.jsx`)

```
지문 입력 → AI가 5개 진술문 + O/X 조합 5지선다 생성
→ 화면에 결과 표시
```

- **AI 사용**: 진술문 생성
- **저장 대상 데이터**: `passageText`, `difficulty`, `questionCount`, `questions`

#### (5) 지문 변형 (`/variation` — `PassageVariation.jsx`)

```
기출문제 이미지/텍스트 → OCR로 3파트 추출(지문, 문제, 선택지)
→ AI가 지문을 새 이야기로 변형 (문법골격 유지, 2-Pass 검증)
→ 변형된 텍스트 화면 표시
```

- **AI 사용**: 추출 + 변형 + 검증 (가장 복잡)
- **저장 대상 데이터**: `sourceImages`, `sourceText`, `extractedOriginal`, `extractedQuestion`, `extractedOptions`, `transformedPassage`, `difficulty`

### 2-2. 현재 상태 관리 구조

**`AppContext.jsx`** — 4개의 독립된 state 객체:

| State | 관리 대상 | setter |
|-------|----------|--------|
| `typeBState` | 어휘 기반 문제 전체 | `setTypeBState` |
| `typeAState` | 지문 기반 문제 전체 | `setTypeAState` |
| `readingOXState` | 독해 O/X 전체 | `setReadingOXState` |
| `variationState` | 지문 변형 전체 | `setVariationState` |

**`SettingsContext.jsx`** — API 키(localStorage), 모델(하드코딩)

### 2-3. 현재 한계점

| 한계 | 설명 |
|------|------|
| **데이터 휘발** | 새로고침·페이지 이동 시 입력값·설정·생성 결과 모두 소멸 |
| **히스토리 부재** | 과거 생성 이력을 조회·재활용할 방법이 전혀 없음 |
| **재사용 불가** | 같은 단어 세트로 다른 유형의 문제를 만들려면 처음부터 재입력 |
| **워크플로우 단절** | 생성 → 다운로드 → 종료의 일회성 워크플로우 |

## 3. 히스토리 기능 상세 설계

### 3-1. 데이터 모델 설계

#### Session 공통 스키마

```typescript
interface Session {
  id: string;                  // UUID v4
  type: SessionType;           // 'typeB' | 'typeA' | 'readingOX' | 'variation' | 'cumulative'
  title: string;               // 자동 생성 + 사용자 수정 가능
  createdAt: string;           // ISO 8601
  updatedAt: string;           // ISO 8601
  metadata: SessionMetadata;
  input: SessionInput;         // 유형별 입력 데이터
  settings: SessionSettings;   // 유형별 설정값
  result: SessionResult;       // 유형별 생성 결과
}

interface SessionMetadata {
  model: string;               // 사용한 AI 모델명
  wordCount?: number;          // 단어 수 (해당 시)
  questionCount?: number;      // 생성된 문제 수
  version: number;             // 스키마 버전 (마이그레이션용)
}
```

#### 유형별 Input / Settings / Result 스키마

**(1) typeB (어휘 기반)**

interface TypeBInput {

### 3-2. 저장소 아키텍처

#### Repository Pattern 인터페이스

```typescript
interface SessionRepository {
  save(session: Session): Promise<void>;
  getById(id: string): Promise<Session | null>;
  getAll(options?: {
    type?: SessionType;
    sortBy?: 'createdAt' | 'updatedAt';
    order?: 'asc' | 'desc';
    limit?: number;
    offset?: number;
  }): Promise<Session[]>;
  update(id: string, partial: Partial<Session>): Promise<void>;
  delete(id: string): Promise<void>;
  deleteAll(): Promise<void>;
  count(type?: SessionType): Promise<number>;
}
```

이 인터페이스를 통해 저장소 구현체를 교체하더라도 상위 레이어의 코드 변경 없이 동작한다.

#### IndexedDB 구현 (`IndexedDBRepository`)

```
Database: "english_question_generator"
  └── Object Store: "sessions"
        ├── keyPath: "id"
        └── indexes:
              ├── "type" (유형별 필터)
              ├── "createdAt" (정렬)
              └── "updatedAt" (정렬)
```

- **선택 이유**: localStorage (5MB 제한)는 이미지 Base64 데이터를 저장하기에 부족. IndexedDB는 브라우저별로 수백 MB~GB 단위까지 저장 가능
- **라이브러리**: `idb` (IndexedDB 래퍼, ~1.2KB gzip) 사용 권장 — Promise 기반 API로 코드 간결화

#### 향후 백엔드 확장 시 마이그레이션 전략

```
Phase 1 (현재):  App ──► SessionRepository ──► IndexedDBRepository

Phase 2 (향후):  App ──► SessionRepository ──┬► IndexedDBRepository (로컬 캐시)
                                             └► APIRepository (서버 동기화)
```

- `APIRepository` 구현체를 추가하고, `SyncRepository` 래퍼에서 로컬-서버 간 동기화 로직 처리
- 핵심: **Repository 인터페이스만 준수하면 상위 코드 변경 없음**
- 오프라인 지원: 로컬 저장 후, 온라인 복귀 시 서버로 push (Conflict Resolution은 "Last Write Wins")

### 3-3. 상태 관리 변경사항

#### 신규: `HistoryContext`

기존 `AppContext`와 `SettingsContext`는 그대로 유지하고, 히스토리 전용 Context를 추가한다.

```typescript
interface HistoryContextValue {
  // 세션 목록
  sessions: Session[];
  isLoading: boolean;

  // 필터
  filterType: SessionType | 'all';
  setFilterType: (type: SessionType | 'all') => void;

  // CRUD
  saveSession: (session: Omit<Session, 'id' | 'createdAt' | 'updatedAt'>) => Promise<string>;
  loadSession: (id: string) => Promise<void>;   // → 해당 페이지로 이동 + 상태 복원
  deleteSession: (id: string) => Promise<void>;
  updateTitle: (id: string, title: string) => Promise<void>;

  // 현재 활성 세션
  activeSessionId: string | null;

  // 목록 갱신
  refreshSessions: () => Promise<void>;
}
```

#### 기존 AppContext 수정 방안

기존 `AppContext`의 state setter(`setTypeBState` 등)는 그대로 유지한다. `HistoryContext`의 `loadSession`이 호출되면:

1. 저장된 세션 데이터를 IndexedDB에서 읽음
2. 해당 유형의 state setter를 호출하여 AppContext 상태를 복원
3. React Router의 `navigate()`로 해당 페이지로 이동

```
loadSession(id)
  │
  ├─ repository.getById(id)
  ├─ session.type === 'typeB' → setTypeBState({ ...session.input, ...session.settings, ...session.result })
  ├─ session.type === 'typeA' → setTypeAState(...)
  ├─ ...
  └─ navigate(typeToRoute[session.type])
```

#### 신규: `useAutoSave` Hook

각 페이지 컴포넌트에서 사용할 자동 저장 Hook:

```typescript
function useAutoSave(type: SessionType) {
  // 문제 생성 완료 시점에 자동 저장 트리거
  // 이미 저장된 세션이면 update, 아니면 새 세션 create
}
```

#### 세션 복원 플로우

```
사용자가 히스토리 항목 클릭

  ├─ 해당 유형의 페이지로 navigate
  └─ 사이드바에서 해당 세션을 "활성" 표시 (하이라이트)

복원된 세션에서 "재생성" 버튼 클릭
  │
  ├─ 현재 입력 데이터 + 설정값 유지
  ├─ 기존 questions 초기화
  ├─ AI API 재호출 (새로운 문제 생성)
  ├─ 새로운 결과로 questions 업데이트
  └─ 새 세션으로 자동 저장 (기존 세션은 보존)
```

### 3-5. 자동 저장 로직

#### 저장 트리거 시점

| 유형 | 트리거 시점 |
|------|------------|
| typeB | `generateAllQuestions()` 완료 후 |
| typeA | `generateTypeAQuestions()` 완료 후 |
| readingOX | `generateReadingOXQuestions()` 완료 후 |
| variation | 지문 변형(2-Pass 검증) 완료 후 |
| cumulative | Word 파일 내보내기 완료 후 |

#### 중복 방지 전략

- 세션에 `activeSessionId`를 두어, 동일 세션 내 재생성 시 **새 세션으로 저장** (기존 세션 덮어쓰지 않음)
- 동일 입력으로 생성 버튼을 연타해도 각각 별도 세션으로 저장
- 빈 결과(생성 실패/에러)는 저장하지 않음

### 3-6. 용량 관리 정책

#### IndexedDB 용량 추정

| 데이터 유형 | 예상 크기 |
|------------|----------|
| 텍스트 기반 세션 (단어+결과) | 5~50 KB |
| 이미지 포함 세션 (OCR 입력) | 200 KB ~ 2 MB |
| 지문 변형 (이미지 다수) | 1~5 MB |

#### 세션 상한 및 정리 정책

- **기본 상한**: 100개 세션
- **상한 초과 시**: 가장 오래된 세션부터 자동 삭제 (FIFO)
- **사용자 알림**: 90개 도달 시 "오래된 세션이 곧 자동 삭제됩니다" 안내
- **수동 관리**: 개별 삭제 + "전체 삭제" 버튼 (확인 다이얼로그 포함)
- **이미지 최적화**: 저장 전 이미지를 적절한 해상도로 리사이징하여 용량 절감 (최대 1920px 폭)

## 5. 확장성 고려사항

### 5-1. 백엔드 동기화 시 변경 범위

Repository Pattern을 사용하므로 변경이 최소화된다:

| 구분 | 변경 필요 | 변경 불필요 |
|------|----------|------------|
| 저장소 | `APIRepository` 신규 구현 | `IndexedDBRepository` (로컬 캐시로 유지 가능) |
| 동기화 | `SyncManager` 신규 구현 | `HistoryContext` 인터페이스 |
| 인증 | 로그인/토큰 관리 추가 | 페이지 컴포넌트, UI 컴포넌트 |

### 5-2. 다중 기기 지원 전략

```
기기 A (브라우저)                           기기 B (브라우저)
  │                                          │
  ├─ IndexedDBRepository (로컬)              ├─ IndexedDBRepository (로컬)
  │                                          │
  └──────────► API Server ◄──────────────────┘
                  │
              PostgreSQL
```

- **Conflict Resolution**: Last Write Wins (updatedAt 기준)
- **동기화 주기**: 앱 시작 시 + 세션 저장/삭제 시 push
- **오프라인 지원**: 로컬 IndexedDB에서 즉시 동작, 온라인 복귀 시 일괄 sync

### 5-3. 데이터 마이그레이션 방안

세션 스키마에 `metadata.version` 필드를 포함한다:

```javascript
// sessionRepository.js 내 DB 오픈 시
const db = await openDB('english_question_generator', DB_VERSION, {
  upgrade(db, oldVersion, newVersion) {
    if (oldVersion < 1) {
      // v1: 초기 스키마
      const store = db.createObjectStore('sessions', { keyPath: 'id' });
      store.createIndex('type', 'type');
      store.createIndex('createdAt', 'createdAt');
      store.createIndex('updatedAt', 'updatedAt');
    }
    if (oldVersion < 2) {
      // v2: 향후 스키마 변경 시 마이그레이션 로직
    }
  }
});
```

## 6. 위험 요소 및 대응 방안

| 위험 요소 | 영향 | 대응 방안 |
|----------|------|----------|
| **IndexedDB 브라우저 호환성** | Safari 시크릿 모드에서 IndexedDB 제한 | 저장 실패 시 사용자에게 안내 메시지 표시 + 정상 모드 사용 권고 |
| **대용량 이미지 저장 시 성능** | 이미지 Base64가 세션당 수 MB | 저장 전 이미지 리사이징 (최대 1920px), 썸네일 별도 생성 |
| **IndexedDB 용량 한계** | 브라우저별 상이 (Chrome ~무제한, Safari ~1GB) | 100개 세션 상한 + 용량 모니터링 + 자동 정리 |
| **기존 코드와의 충돌** | AppContext setter 호출 시점 변경 | 기존 setter 인터페이스 유지, 히스토리 로직은 별도 레이어에서 처리 |
| **새로고침 시 활성 세션 상태 유실** | URL만으로는 어떤 세션을 보고 있었는지 알 수 없음 | `activeSessionId`를 `sessionStorage`에 저장, 새로고침 시 자동 복원 |
| **동시 탭 문제** | 여러 탭에서 같은 IndexedDB 접근 | IndexedDB는 트랜잭션 기반이므로 충돌 위험 낮음. 필요 시 BroadcastChannel API로 탭 간 동기화 |
